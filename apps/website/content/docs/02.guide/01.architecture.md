# Architecture

Inkline is a compiler, not a runtime. A single `.ink.tsx` source is parsed into a
language-agnostic intermediate representation (IR), analyzed once, then emitted
per target. The frameworks never share a runtime — each gets idiomatic,
standalone output.

## The pipeline

```
.ink.tsx  ──parse──▶  IRModule  ──analyze──▶  AnalyzedModule  ──emit──▶  Code IR  ──print──▶  files
                      (language-agnostic)                      (per target)         (per target)
```

The IR is the architectural pivot. Parsing and analysis are done **once** and are
target-agnostic; only the final emit and print stages are per-framework. That is
what keeps seven targets consistent: they all descend from the same analyzed
module.

## Why an IR

The IR records structure and reactivity independently of any framework's syntax:

- **`IRModule`** — a parsed file: its components, contexts and imports.
- **`IRComponent`** — a component's props, slots, events, state and render tree.
- **`IRExprNode`** — a single expression plus the reactive dependencies it reads.
  The expression is preserved verbatim; codegen rewrites it per target.

Reactivity is tracked with a symbol table: each signal, memo, prop or ref carries
a stable identity, and every expression records which symbols it reads. This
dependency graph is what lets each target wire up fine-grained reactivity in its
own idiom.

## One IR, many targets

Each framework is a small `Target`: a set of rewrite rules plus an `emit`
function. Parsing, analysis and the IR are reused unchanged. The rewrite rules
capture how a framework expresses a reactive read, for example:

- **React** — `count()` becomes `count`; setters are function calls; attributes
  are camelCased.
- **Vue** — `count()` becomes `count.value`; setters are field assignments;
  attributes and events are kebab-cased.
- **Svelte** — direct variable access, using runes for state.

Adding a framework is a new `Target` plus an `emit` function — the pipeline in
front of it does not change.

## Headless and styled components

Components are authored in two layers. **Headless** components define structure,
slots, props and events with no styling. **Styled** components wrap a headless
one and apply design-token classes. Consumers of a framework package import the
styled variant by default; the headless variant is available for fully custom
styling.

See the [Button](/docs/components/button) page for a styled component rendered
live across every framework, and [Getting Started](/docs/getting-started/installation)
to compile your own.
